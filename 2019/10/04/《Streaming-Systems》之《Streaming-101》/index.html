这是《Streaming Systems》系列的第一篇。

《Streaming 101》讲了什么

作为介绍 Streaming Systems 的开篇，自然会先介绍什么是流式系统，然后说明流处理引擎为了超过批处理引擎应该做到什么，之后讲到了 Event Time 和 Processing Time 的区别，最后以常见的数据处理模式作为结尾。

流式系统是什么？

Tyler Akidau 大神给出的定义是：一种为了处理无穷数据集的数据处理引擎。关于数据集，大神又给出了两种不同的维度进行衡量。一种是从基数（cardinality），另一种就是构造（consitution）。

从基数角度来说，数据集可以分成有界或者无界两种形式。有界数据集自然是数据量有限的数据集，比如说给你一批订单，让你求出这批订单的总量，这里的订单数量就是有限的，而无限数据集则是像求出实时热门商品，这时就会有无穷无尽的商品信息从源头发来。

而从构造角度来说，可以分成表（Table）和流（Stream）两种结构。SQL 系统处理的数据集就是表结构的。而流结构的数据集则是 Map-Reduce 家族经常处理的。

在某种程度上，有界数据集可以看成无界数据集的一种特殊情况，而表结构和流结构又可以通过时间联系起来（这个会在未来的文章中提到）。所以总得来说，流式系统可以处理各种各样的数据，不过在处理有界数据集的时候，可能效率会不如专门为这种数据集设计的系统。

流式处理引擎怎样才能代替批处理引擎？

在这篇文章发表的时候也就是2015年，那时候还流行Lambda架构，这种架构大体上就是流式引擎和批引擎同时运行，流式引擎提供低延迟但不怎么准确的结果，而批引擎的结果会慢一点但是准确的。

不难看出，流式引擎要能和批引擎平起平坐的一个关键点就是准确性（correctness），而准确性的核心就是一致性状态的存储。就像 Flink 那样可以通过堆内存等方式对 checkpoint 进行存储。做到这一点的先驱则是 spark streaming 。如果流式引擎做到了强一致性，那么它就可以做到对数据准确一次（exactly-once）的处理。对于准确一次这里简单提一下，对于数据处理的级别可以大体分为三类，分别是最多一次（at-most-once）、最少一次（at-least-once）以及准确一次。不难看出，准确一次的实现，可以让数据不会重复且能确保被处理到，所以这是数据处理引擎能实现的最好的级别。

而流式引擎想要战胜批引擎，还需要一种工具，那就是能推导数据的产生时间，而这就是下文所讲述的重点。

Event Time 和 Processing Time 的区别

对于无限的数据流和数据处理系统来说，有两个时间角度是经常需要关注到的，分别是 Event Time 和 Processing Time。先来看下 Event Time 和 Processing Time 的定义。

Event Time ：事件发生时的时间

Processing Time ：事件被系统处理的时间

最理想的情况自然是 Event Time 和 Processing Time 两者相同，但可惜的是现实情况两者往往是不同，比如网络延迟、上游处理导致的延迟等等，比较极端的还可能是比如你在坐飞机，然后把手机调成了飞行模式，那很多数据只能等到你下飞机才能发送，这时候 Event Time 和 Processing Time 的差别就更大了。下图就可以反映两者的区别。

Event Time 和 Processing Time 其实是从两个不同角度来对数据进行观察的。但相对而言，Event Time 更重要些，因为它反映的是现实时间，毕竟我们做各种决策肯定是在现实这个角度，而不是机器的时间角度。

为了处理无限的数据，一种简单的方法就是把数据流按照某个原则进行切割，把无限的数据变成一批批有限的数据。那么问题就来了，对于当时的系统来说，它们是以 Processing Time 的角度来进行数据切割的，而你关注的是 Event Time，那么就会产生一些数据被分配到不正确的批中了。所以如果一个系统能推导数据的 Event Time 那自然就不会产生上述问题。

可 Event Time 就那么美好么？以上面用时间为原则来对数据集进行切割为例，如果用 Processing Time ，那么系统很容易就知道什么时候该对这批数据进行计算，只要系统时间到了规定时间就行。可是从 Event Time 的角度呢？还记得数据集是什么样的么？这是一个无限且乱序的数据集，所以对于系统而言，它永远也无法保证这批数据完全包含了应该包含的数据。额外提一句，Flink 使用 watermark 解决这个问题，如果有兴趣可以看看相关文档。

下面讨论下常见的数据处理模式（data processing pattern）

数据处理模式

有限数据

这种数据集的处理方式很简单，就用通常的批处理引擎，比如 MapReduce 就可以很快算出结果了。

用批引擎处理无限数据

尽管批处理引擎不是为了处理无限数据设计的，不过我们可以从中看出他处理这类数据的不足，流处理引擎从而可以以之为鉴。对于无限的数据集，批处理引擎自然也是通过把无限的数据切割成一批一批的有限的数据集，再通过自己丰富的经验处理它:)。

固定的 Window

最经典的方法就是把数据集按照固定时间进行切割。就像下图所示，把数据按照10:00 - 11:00这样的时间段分成不同的批次来进行处理。

但这样带来的问题就像之前讲过的，怎么能确保一个窗口的数据是完全的呢？

会话的情况（Session）

这种切割方式在处理会话的时候，问题就更严重了。所谓会话就是用户在一段时间内的活跃行为。按照下图所示，这种切割方式可能会造成用户同一个会话的数据被切割到不同的时间段了，这肯定是不允许的。

用流引擎处理无限数据

Tyler Akidau 大神把流引擎处理无限数据分为四大类：1.时间无关（Time-agnostic）2.近似计算（approximation）3.通过process time作为窗口 4.通过event time作为窗口 接下来将会依次介绍这四种情况。

时间无关的

在这种情况下，只需要关注数据本身，对于数据所附带的时间因素则无需关注，因此批处理引擎其实也可以完美解决这种情况。

一种典型的场景就是数据过滤了,比如你要统计一个商品的购买总数，那只需要把不是这种商品的数据过滤掉就好。

另一种场景就是内联了(Inner Joins)。这种时候就是当一个流里的数据需要和另一个流里的数据进行结合再输出。

近似算法

近似算法就是指求诸如 Top-N 这种场景。这种算法很有可能需要用到时间这一维度，比如求某段时间的购买量最高的商品。

把 processing time 作为窗口

在进行叙述如何用 processing time 作为窗口之前，先讲一下窗口有哪些分类。

第一种窗口就是固定窗口（fixed window），又称为滚动窗口（tumbling window），这种窗口的特点就是窗口大小是固定的，每个窗口之间相互独立。

第二种窗口是滑动窗口（sliding window），相较于第一种窗口而言，两个窗口之间是有重叠的部分。

第三种窗口则是会话窗口（session window），这种窗口是用来分析用户行为的。

如下图所示，数据根据来到系统的顺序被分配到不同的事件窗口之中。

按照 processing time 生成窗口很简洁明了，就是按照系统的机器的系统时间，所以这种数据处理模式就不需要思考是否有数据延迟的情况，就很适合那种从机器角度考虑的事件，比如计算一个网站每秒中的访问量。

把 event time 作为窗口

在前文中已经叙述过 processing time 的不足之处，就是按照现实世界的角度来说，可能会造成数据被分配到不正确的窗口之中。如下图所示，有两条数据被 processing time 分配到了错误的窗口之中，而在 event time 的情况下就正确了。所以正确性就是 event time 相较于 processing time 的一条优势。

还记得之前说的会话（session）的情况么？在 event time 的情况下就能被很优雅的处理，只要通过建立动态大小的窗口就行了。

可是天下没有免费的午餐，event time 有这么多优势，自然也会有自己的缺陷。第一：由于窗口的生命周期变得更大，所需的内存或磁盘空间也会变得更大。第二：就是之前提到过的如何确认一个窗口已经获取了它所需的所有数据，这点将会在之后进行讨论。
